Feature Management Service 架构设计

100+ 应用 · 10,000+ 开关 · 每秒 10 万次查询（100K QPS）· 99% 请求在 10ms 内返回（P99 <10ms）

三个核心决策：
1. 缓存开关定义（非评估结果），SDK 本地执行 → 调 isEnabled() 时无网络请求
2. 稳定哈希分桶+salt做百分比灰度 → 同用户永远同桶，跨开关无关联
3. 确定性重放替代评估日志 → 存版本历史(~50MB) vs 日志(864GB/天)

0. 背景

电商有新功能想灰度——先给 5% 美国用户试用，没问题再放量。这就是功能开关：后台点一下，线上行为就变，不用部署。

问题：应用问"new_checkout 对用户 U123 是开是关？"→ 10ms 内回答，每秒 10 万次。

1. 核心思路：SDK 本地算

每次 isEnabled() 发 HTTP → 10万QPS全慢5ms，服务器挂了业务全停。配置系统不该有这么大影响范围。

做法：SDK 启动时下载完整快照到内存，之后评估全部本地完成。像手机离线地图。

图 1 — Snapshot 只管下发配置，不参与每次评估
设计灵魂：高频评估走本地，低频的配置下载走服务端。控制面（管理开关的后台操作，低频）和数据面（SDK 评估开关，高频）完全分离。

2. 跨 Region：一条变更怎么到全球 SDK

图 2 — 写只在主区，Kafka 扇出，每区独立部署全链路

关键：写只在 us-east（PG 主库）；每区有独立 Redis + Snapshot API + CDN；Management API 每次写操作后同步发 Kafka 事件（自带完整 flag 定义 + 受影响的 app 列表），Consumer 收到后直接 merge 进 Redis，不需要查 PG（Kafka 自带数据，不依赖跨区 PG 副本）；跨区 Kafka 延迟 1~5s，开关场景可接受。

注：eu-west/ap-southeast 的运营改 flag 时，Management API 在本区接收请求后转发写到 us-east 主库（多 200ms RTT，可接受），读仍走本区。完全 multi-master 配置系统代价远高于收益。

业务开发者只看到一个对象两个方法。里面有三层逻辑加一个旁路同步器。

图 4 — 三层读链路 + 右侧同步控制器写入缓存

移动端额外优化：进程被杀是常态，每次冷启动如果只靠内存缓存就白拉一次完整快照（~1MB）。移动 SDK 把 snapshot + ETag 持久化到本地磁盘，下次冷启动先读本地 ETag 走 If-None-Match，snapshot 没变就 304 空响应，省掉 ~99% 的重复下载。

6. API 设计

本节目录： 6.1 数据面（SDK 拉快照） · 6.2 控制面（运营改 flag） · 6.3 核心模型 · 6.4 rule 里的 op 词汇表 · 6.5 definition 示例 · 6.6 运维 API（/explain）

6.1 数据面（给 SDK）

GET /api/v1/snapshot
  Header: Authorization: Bearer <api_key>     // app 身份从这里解析（api_key 绑应用，签名验证在 7.3 节）
  Header: X-Env: production                    // 必填，指定要哪个环境的快照
  Header: If-None-Match: "v42.7"               // ETag 缓存
  → 200  { "version":"v42.7", "flags":[/* 仅本 app 可见的 flag */], "etag":"v42.7" }
  → 304  (body 为空)

env 怎么传：用 X-Env HTTP header 标明目标环境（生产 SDK 默认 production；staging SDK 在部署时通过环境变量切换）。不用 query 参数——避免 CDN 把同一 URL 当成不同 cache key 浪费命中率。

app 怎么传：不在请求里传，从 api_key 反查。每个 api_key 在 applications 表里绑死一个 app，请求进来先验证 api_key 拿到 app_id，再用 app_id 过滤可见 flag。这样 SDK 代码不用关心自己属于哪个 app，运营也不用在 SDK 部署时配置 app 名。

一个接口，返 本 app 可见的最新快照（典型 50 条 flag，不是全量 10k 条）。走 CDN，ETag 缓存（类似浏览器缓存：服务端返回版本号，SDK 下次请求带这个号，没变化就回 304 空响应，有变化才传数据）。

实现就一层薄薄的读：

// Snapshot API — 伪代码，实际就这些
func HandleSnapshot(w, r):
    api_key  = r.Header["Authorization"][len("Bearer "):]
    app      = authenticate(api_key)                        // HMAC 校验 + 拿到 app_id
    env      = require(r.Header["X-Env"])
    key_data = fmt.Sprintf("snapshot:%s:%d:data", env, app.id)   // 每个 (env, app) 一份
    key_ver  = fmt.Sprintf("snapshot:%s:%d:version", env, app.id)

    // ETag 客户端带引号（HTTP spec），Redis 存的不带——必须 Trim 才能比对
    if strings.Trim(r.Header["If-None-Match"], "\"") == redis.Get(key_ver):
        w.WriteHeader(304)
        return
    flags = redis.Get(key_data)
    w.Write(JSON{version: redis.Get(key_ver), flags: flags, etag: redis.Get(key_ver)})

关键：version 和 data 必须原子写。如果分两次 SET，读端可能看到新 version + 旧 data（或反过来），SDK 缓存状态撕裂。用 Lua 在 Redis 端一次写两个 key：

-- Consumer 将单个 flag merge 进 per-app snapshot（原子 INCR + SET）
-- KEYS[1]=version key, KEYS[2]=data key, ARGV[1]=json
redis.eval([[
    local v = redis.call('INCR', KEYS[1])
    redis.call('SET', KEYS[2], ARGV[1])
    return v
]], {key_ver, key_data}, {json_data})

方案 B — Kafka 自带完整数据。Management API 写 flag 后同步发 Kafka 事件：{ flagKey, env, op, definition, scopedAppIds }。每个 Region 的 Consumer 收到事件后，直接把 flag merge 进每个关联 app 的 Redis snapshot——不需要查 PostgreSQL。Lua 脚本原子地递增版本号（INCR）并写入新快照（SET）。60s @Scheduled 全量 sweep 作为兜底，覆盖 Kafka 丢失的极端情况。每次 flag 变更是 N 次 Redis 写入（每个关联 app 一次），而不是 1 次——这是 per-app 隔离的写放大代价，换来了 CDN 按 app 缓存、SDK 小带宽拉取、跨团队 flag key namespace 隔离的能力。

一个具体例子（checkout-service 这个 app 看到的）：

// GET /api/v1/snapshot   Authorization: Bearer sk_live_xxx   X-Env: production   →   200 OK
{
  "version": "v42.7",
  "etag": "v42.7",
  "flags": [
    { "key":"new_checkout",    "definition":{ "type":"pct_rollout","pct":20,"salt":"a3f9",... }},
    { "key":"vip_discount",    "definition":{ "type":"boolean" }},
    { "key":"admin_debug",     "definition":{ "type":"targeting","rules":[...] }}
    // 仅本 app 可见的 50 条，不是全量 10k 条
  ]
}

就一份过滤后的 JSON，gzip 压缩后约 20KB–80KB（取决于本 app 可见 flag 数和 rules 复杂度）。不含 version、state、owner、updated_at 这些管理字段——SDK 只需要 key + definition 就够了，state 在 DB 列里管，snapshot 不带。

CDN 还要不要：要。CDN cache key 包含 app（解析自 api_key），缓存 (env, app) 组合。每个 cache key 的请求量从原来单 key 的 35M-50M/天 摊到 N 个 key，热门 app 命中率仍 95%+，冷门 app 命中率低。100 个 app 时总命中率 ~85%，可以接受；1000+ app 时需要分层（只对热门 app 走 CDN 缓存，冷门 app 直连 origin）。

6.2 控制面（给后台管理）

POST   /api/v1/flags                       创建
PUT    /api/v1/flags/{key}                 更新（version 乐观锁，请求必须带 If-Match: "<version>"，更新时检查 version 是否还是旧值，防止两个运营同时改同一个开关互相覆盖）
GET    /api/v1/flags                       搜索（分页：?cursor=&limit=）
GET    /api/v1/flags/{key}                 详情 + 历史
DELETE /api/v1/flags/{key}                 软删除（state 置为 archived，snapshot 不再下发；保留 flags_history 用于 explain）
POST   /api/v1/flags/{key}/rollout         灰度：0%→10%→50%→100%（专门改 pct）
POST   /api/v1/flags/{key}/targeting       改 rules 数组（专门改定向条件）

rollout 不是什么复杂的状态机，就是一次普通的 flag 更新，只不过专门改 pct：

// POST /api/v1/flags/new_checkout/rollout   body: {"pct": 50, "expected_version": 41}

BEGIN;
  UPDATE flags SET version = version + 1,
      definition = jsonb_set(definition, '{pct}', '50'::jsonb),  -- 必须 ::jsonb cast，否则存成字符串
      updated_at = now()
  WHERE key = 'new_checkout' AND env = 'production' AND version = 41;
  -- 0 rows affected → 返回 412 Precondition Failed

  INSERT INTO flags_history (flag_key, env, version, definition, changed_by)
  VALUES ('new_checkout', 'production', 42,
          (SELECT definition FROM flags WHERE key='new_checkout'), 'alice');
COMMIT;

本质上和 PUT /api/v1/flags/{key} 做的事一样——乐观锁更新 + 写历史。单独拆一个 /rollout 只是为了让运营操作更直观，不用懂整个 JSON 结构。

6.3 核心模型

-- DB flags 行（state/version 走列，不进 JSONB definition）
key="new_checkout_flow", env="production", version=42, state="active",
critical=false, owner="checkout-team", release="v3.2.0",
definition = {                              -- JSONB
  "type":"pct_rollout", "pct":20, "salt":"a3f9c2e1",
  "rules":[
    {"attr":"region","op":"in","values":["us-east-1"]},
    {"attr":"tenant","op":"eq","value":"enterprise"}
  ]
}

rules 为 AND 语义，全部满足后 pct 生效，即"满足规则的人中的 pct%"。state 和 version 在 DB 列管，不写进 definition JSON——避免双写导致数据漂移。

6.4 rule 里的 op 词汇表

SDK 必须支持完整 op 集合；运营在管理后台写规则时只能选这些，不能自由输入
op含义示例
eq / ne等于 / 不等于{"op":"eq","value":"enterprise"}
in / not_in属于 / 不属于集合{"op":"in","values":["us-east-1","us-west-2"]}
gt / gte / lt / lte数值比较{"op":"gte","value":18}
contains字符串包含{"op":"contains","value":"premium"}
starts_with字符串前缀{"op":"starts_with","value":"ios_"}
regex正则（用 RE2 语法，避免 ReDoS）{"op":"regex","value":"^vip_.*"}
exists属性非空{"op":"exists"}

SDK 启动时校验：遇到不认识的 op 直接 fail-fast，不静默忽略——否则 5% 的 flag 用新 op 但 SDK 旧，会被静默关掉。

6.5 definition 示例：三种典型开关

① 全站开关——是否启用看 DB 列 state，active 即 true，archived 即 false。definition 里只标 type。

{ "type": "boolean" }

② 百分比灰度——"10% 用户看到"，salt 保证各开关的灰度用户互相独立

{ "type": "pct_rollout", "pct": 10, "salt": "a3f9c2e1" }

③ 定向规则——"只给 admin 角色看"

{
  "type": "targeting",
  "rules": [{ "attr": "role", "op": "eq", "value": "admin" }]
}

④ 混合——规则过滤 + 百分比，最常用的形式

{
  "type": "pct_rollout",
  "pct": 20,
  "salt": "a3f9c2e1",
  "rules": [
    { "attr": "region",    "op": "in", "values": ["us-east-1"] },
    { "attr": "tenant",    "op": "eq", "value": "enterprise" },
    { "attr": "beta_user", "op": "eq", "value": true }
  ]
}

条件匹配器就是一段几十行的 if/else：先跑 rules（AND 语义，一条不满足就返回 false），全过了再按 type 处理——优先级见上面的"type × rules × pct"真值表。

6.6 运维 API（/explain）

用户报障说"昨天 14:00 我没看到新收银台"——运营拿这个 API 一查就能给用户解释。

API 长这样（用就是这一行）：

GET /api/v1/explain?flag=new_checkout&user=U123&at=2026-06-24T14:00:00Z

→ 200
{
  "enabled": false,                              ← U123 当时看到的
  "reason":  "用户 bucket=15，但 14:00 时 flag 是 10% 灰度，15 ≥ 10 → false",
  "flag_version": 41,                            ← 当时用的是哪个版本
  "user_context": { "region": "us-west-2", "tenant": "enterprise" }   ← 当时 U123 的资料
}

怎么算出来的：不存"当时是 true 还是 false"（100K QPS 一天 864 GB，存不起）。只存 flag 每次被改的版本（一天顶多几百次）。问的时候拿当时那条 flag 定义 + 当时那个用户的资料重新跑评估——结果跟当时一模一样，因为评估是纯本地算的（hash 桶号 + 规则 AND），同样的输入永远出同样的输出。整个 history 表 7 年下来也就 50–500 MB。

唯一搞不定的：flag 的 salt 后来被改过——重算用的盐已经不是当时那个，桶号会变。这时返回"无法精确回答，去 audit_log 查"。

图 6 — 不存每次结果，只存 flag 改动，问的时候重算

7. 数据库设计

CREATE TABLE flags (
    key         VARCHAR(128),
    env         VARCHAR(16) NOT NULL DEFAULT 'production',
    version     INTEGER NOT NULL,               -- 乐观锁
    state       VARCHAR(16) NOT NULL DEFAULT 'active',  -- active | archived（不进 JSONB definition）
    definition  JSONB NOT NULL,                 -- 只有 type/pct/salt/rules
    critical    BOOLEAN NOT NULL DEFAULT false,
    owner       VARCHAR(64),
    updated_at  TIMESTAMP NOT NULL DEFAULT now(),
    PRIMARY KEY (key, env),
    CHECK (env IN ('dev','staging','production','preprod')),
    CHECK (state IN ('active','archived'))
);

-- flags_history：复合主键 + FK 引用 flags，避免并发写重复行和历史漂移
CREATE TABLE flags_history (
    flag_key    VARCHAR(128) NOT NULL,
    env         VARCHAR(16) NOT NULL,
    version     INTEGER NOT NULL,
    definition  JSONB NOT NULL,
    changed_by  VARCHAR(64),
    changed_at  TIMESTAMP NOT NULL DEFAULT now(),
    PRIMARY KEY (flag_key, env, version),       -- 一个 (flag, env) 的每个 version 只能一行
    FOREIGN KEY (flag_key, env) REFERENCES flags(key, env) ON DELETE CASCADE
);
CREATE INDEX idx_hist_time ON flags_history(flag_key, env, changed_at DESC);
-- flags_history 不删：audit 要求保留 ≥7 年；超过 1 年的冷数据按月 partition + 归档到 S3

改开关时一个事务：乐观锁更新 flags + INSERT flags_history。FK 兜底——运营 typo 出错（比如 key 多打一个字母）会直接报 FK violation，不会留下孤儿历史行。

环境怎么处理？

同一个开关在 staging 放 50%、production 还是 0%，不同环境值不同。做法很简单：把 env 加到 flags 表主键里，一行 = 一个开关在一个环境里的配置。4 个环境（dev/staging/preprod/production），每个开关最多 4 行。

Flag → App 可见范围

-- 每个 flag 在每个 env 下显式列出可见的 app；不在这张表里的 (flag, env, app) 默认不可见
CREATE TABLE flag_app_scopes (
    flag_key  VARCHAR(128) NOT NULL,
    env       VARCHAR(16) NOT NULL,
    app_id    INTEGER NOT NULL REFERENCES applications(id) ON DELETE CASCADE,
    PRIMARY KEY (flag_key, env, app_id)
);
CREATE INDEX idx_scope_app ON flag_app_scopes(app_id, env);

作用：运营创建/改 flag 时勾选哪些 app 可见（"checkout-service 和 payment-service 能看，其他都看不到"）。Snapshot API 按 api_key 解析出 app_id，JOIN 这张表过滤可见 flag。10k flag × 100 app × 4 env = 4M 行，多对多表完全撑得住。

顺带解决了 flag key 重名问题：两个 team 各自取名 new_ui，只要它们的 flag 绑不同 app，snapshot 不会冲突。原先要求 flags PK 是 (key, env) 的硬性约束现在更宽松——key 重名但 app 不同的两个 flag 互不影响。

应用 + 凭证

-- 服务级身份：SDK 用 api_key 调 Snapshot API
-- user-level 身份：走 SSO/OIDC（不是这张表管），audit_log.actor 来自 user 层
CREATE TABLE applications (
    id            SERIAL PRIMARY KEY,
    team          VARCHAR(64) NOT NULL,        -- 所属团队（多团队隔离命名）
    name          VARCHAR(64) NOT NULL,        -- "checkout-service"
    env_scope     VARCHAR(16) NOT NULL DEFAULT 'production',  -- key 绑 env，限制误用
    api_key_hash  VARCHAR(255) NOT NULL,       -- bcrypt/argon2 哈希，不存明文
    api_key_prefix VARCHAR(16) NOT NULL UNIQUE,-- 存 key 前 8 位方便运维识别（不存全 key）
    owner         VARCHAR(64),
    created_at    TIMESTAMP NOT NULL DEFAULT now(),
    UNIQUE (team, name),                       -- 避免不同 team 撞同名 service
    CHECK (env_scope IN ('dev','staging','production','preprod'))
);

为什么存哈希：明文存 api_key → 任何能 SELECT 这张表的人都拿走所有服务凭证 → 任一 service 被攻破全部沦陷。bcrypt 哈希后比对慢（~100ms），但 Snapshot API 路径可以加内存 cache 兜性能。

API Key 怎么用（HMAC 哈希）

API key 是 256 位高熵随机串，HMAC-SHA256(server_secret, raw_key) 哈希后存 api_key_hash。DB dump 出去也还原不出明文，api_key_hash 列加 UNIQUE 直接按 hash 查找。server_secret 单独存 Vault，运行时注入。

注册（一次性）：运营通过内部管理后台（带 SSO 登录）调用 POST /api/v1/admin/applications，body 是 (team, name, env_scope)，不用脚本，也不在外网开放。

// register-application(team, name, env_scope)
raw_key = "sk_live_" + crypto_random(24)              // 32 字节明文，只显示这一次
hash    = HMAC-SHA256(SERVER_SECRET, raw_key)         // μs 级
INSERT INTO applications (team, name, env_scope, api_key_hash, ...)
  VALUES (?, ?, ?, hash, ...);
return raw_key     // 运营存进 vault，DB 再无明文

每次请求验证：

// Snapshot API 入口
submitted = r.Header["Authorization"][len("Bearer "):]   // 完整 key
hash      = HMAC-SHA256(SERVER_SECRET, submitted)
row       = SELECT id, env_scope, revoked_at
            FROM applications WHERE api_key_hash = hash  -- UNIQUE 索引，O(1)
if row is null:                       return 401
if row.revoked_at != null:            return 401         -- 撤销
if row.env_scope != requested_env:    return 403         -- key 绑 env，越权
// 验证通过

旋转 / 撤销：

-- 运营主动 rotate
UPDATE applications SET revoked_at = now() WHERE id = ?;        -- 老 key 立即死
INSERT INTO applications (..., api_key_hash)                    -- 新 key 入库
  VALUES (..., HMAC-SHA256(SERVER_SECRET, new_raw_key));
-- SDK 收到 401 后去 vault 读新 key，零停机

撤销后老的 hash 仍在表里，靠 revoked_at 过滤；rotate 就是改 revoked_at + 插新行。

审计日志

CREATE TABLE audit_log (
    id          BIGSERIAL PRIMARY KEY,
    actor       VARCHAR(64) NOT NULL,                 -- 来自 user auth (SSO/OIDC)，不是 api_key
    action      VARCHAR(32) NOT NULL,                 -- 见下方 CHECK
    target      VARCHAR(128) NOT NULL,                -- flag_key 或 (app team/name)
    detail      JSONB,                                -- 变更细节（体积限 8KB）
    ip_address  INET,
    created_at  TIMESTAMP NOT NULL DEFAULT now(),
    CHECK (action IN ('created','updated','rolled_out','targeted','archived','queried','login'))
);
CREATE INDEX idx_audit_actor_time ON audit_log(actor, created_at DESC);
CREATE INDEX idx_audit_target     ON audit_log(target, created_at DESC);

两层身份必须分清：api_key 是"哪个服务在调"（machine auth），actor 是"哪个人在操作"（user auth）。一个 service key 不能伪装成"alice"——actor 由 SSO/OIDC 注入，request 走 JWT 带 sub claim 进来，audit 不会被服务被攻破而污染。

保留多久：两类分开——改 flag 的记录留 7 年（财务审计要查这么久），"谁查了 flag" 这种读操作的记录 30 天后转到 S3（量太大，DB 存不起）。

怎么查：出事后想知道谁干的，按"操作人 + 时间"或"改的是哪个 flag + 时间"过滤就够。

紧急回滚：核心 flag 被误推到 100% 时，先 UPDATE flags SET state='archived'（snapshot 秒级不再下发这个 flag，archive 不等于 delete，历史还在），再走 /rollout 回滚到上一个版本。

9. 关键决策总结

关键决策及代价（每行都是权衡）
决策为什么放弃什么
缓存定义，不缓存结果调 isEnabled() 时无网络请求，管理后台挂了，SDK 照样用本地缓存在评估每实例 2MB 内存
Push + Poll 双通道紧急秒级生效，普通低成本不保证所有 SDK 同一时刻看到完全相同的配置（最坏 60s 延迟）
哈希分桶 + salt纯本地，桶独立不关联换算法需重部署
确定性重放50–500MB 一次性 vs 864GB/天日志不含用户上下文变化（用 1% 采样兜底）
Per-app Snapshot + CDN每个 app 只看到自己可见的 flag，顺带解决跨 team flag key 重名；CDN 边缘仍可缓存 (env, app) 维度CDN cache key 数从 4 个变成 4×N 个，100 app 时总命中率 ~85%；1000+ app 需要分层只缓存热门 app

1000+ app 时怎么再演进：CDN cache 命中率降到 70% 以下时，按 app 热度分层——热门 app（前 100）继续走 CDN 全量缓存，冷门 app 直接 origin（origin 加一层进程内 LRU 缓存兜底），避免 N 个冷 cache key 浪费边缘内存。

10. 实现顺序

  1. Flag 模型 + 版本化存储 — PostgreSQL 三张表（flags / flags_history / flag_app_scopes）+ 建表 migration
  2. Management API — 控制面 CRUD（创建/更新/删除/搜索开关）
  3. Snapshot API — GET + ETag，不到 100 行
  4. 条件匹配器 — 规则匹配 + 哈希分桶，不到 200 行
  5. SDK 演示 — 内置定时器轮询 Snapshot + 条件匹配器评估
  6. Explain API — 历史版本查找 + 重放
  7. Push 通道 — Redis Pub/Sub 推变更信号，SDK 收到后拉快照

核心洞察——本地评估 + 确定性重放——几百行代码就能跑通端到端最小可用版本。
